# Task: Task 3.10 — Audit Display "Operationally Clean" Definition

## Context

Task 3.5 cohort-aware baseline audit (commit `c28319d`) surfaced 16 LOCKED_VIOLATION_REPLAY lines on first live run. Codex engineering review (per Claude consultation) recommended Option B: do NOT update lock rows. Instead, improve audit display to define "operationally clean" criteria.

The 16 REPLAY_DRIFT lines = E3 baseline cohort lines also affected by BTP-002 recipe drift (PROD-006/PROD-023). They are informational, not critical. stored_cost_at_sale unchanged — no security incident. They will appear in audit output each run because replay keeps shifting as recipes change. This is expected behavior.

## Goal

Update audit script + supporting library code to:

1. **Define "operationally clean" explicitly**: clean = `LOCKED_VIOLATION_STORED = 0 AND NEW_INVESTIGATION_NEEDED = 0 AND KNOWN_NOT_LOCKED = 0`. LOCKED_VIOLATION_REPLAY is informational and does NOT affect clean status.

2. **Stdout summary**: clear "OPERATIONALLY CLEAN" or "REVIEW REQUIRED" status with structured breakdown:
   ```
   MAC Drift Baseline Audit: OPERATIONALLY CLEAN
   - 380 LOCKED_MATCHED (cohort understood, no action)
   - 16 LOCKED_VIOLATION_REPLAY (informational, known BTP drift pattern)
   - 0 LOCKED_VIOLATION_STORED (no security incident)
   - 0 KNOWN_NOT_LOCKED (all understood lines locked)
   - 0 NEW_INVESTIGATION_NEEDED (no new drift)
   ```

3. **Exit code**:
   - `0` if operationally clean
   - `1` if REVIEW REQUIRED (STORED violation OR new investigation OR known-not-locked)

4. **REVIEW REQUIRED case** — show prominently at top of stdout:
   ```
   MAC Drift Baseline Audit: REVIEW REQUIRED
   Critical: N LOCKED_VIOLATION_STORED (security incident)
   Action: N NEW_INVESTIGATION_NEEDED (new drift to investigate)
   ...
   ```

5. **Policy doc update**: add "Operational clean" section to `docs/audits/2026-07-16-btp-recipe-replay-drift-policy.md` defining the criteria. Reference for future operators.
