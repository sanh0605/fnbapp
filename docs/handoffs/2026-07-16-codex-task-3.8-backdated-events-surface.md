# Task: Task 3.8 — Surface Backdated-Ledger Events for 41 Outside-Cohort Lines

## Context

Task 3.4 (`fea097d`) classified 41 outside-cohort lines as `BACKDATED_LEDGER_LIKE`:
- Precise Task 3.2 PO window fingerprint: `effective_timestamp <= sale_time <= purchase_order.created_at`.
- Total delta: -43,809 VND.
- Disposition per Task 3.4: Task 3.2 admin UI review path.

Task 3.7 (`d2177ca`) locked the other 225 outside-cohort lines (drift cohort). The 41 BACKDATED lines are **excluded** from Task 3.7 cohort because they need operator review via `/admin/audit/backdated-ledger` UI before any decision.

The admin UI exists (`app/admin/audit/backdated-ledger/page.tsx` + detail page + `approveAndRecomputeAction` / `rejectEventAction`). But before the operator walks through, we need to know:
1. Which `backdated_ledger_events` rows underlie the 41 lines.
2. Current status of each event (PENDING / APPROVED / REJECTED).
3. Per-event summary so the operator can decide bulk-approve vs individual review.

## Goal

**Read-only investigation.** Produce a map: 41 line IDs → associated event IDs → status breakdown → per-event summary. **No DB writes. No UI actions. No status mutations.**

## Scope

### In scope

1. Load the 41 BACKDATED_LEDGER_LIKE line IDs from `docs/audits/2026-07-15-task-3.4-outside-cohort-investigation.json`.
2. For each line, find associated `backdated_ledger_events` rows. The mapping may use:
   - `backdated_ledger_events.item_reference` matching the line's consumed items (direct + BTP shortfall decomposition).
   - `backdated_ledger_events.effective_timestamp <= line.sale_date` AND `backdated_ledger_events.visibility_timestamp >= line.sale_date` (Task 3.2 detection window).
   - **Verify the actual mapping logic** used by Task 3.2 detection (read `lib/backdated-ledger/find-affected-lines.ts` and `lib/backdated-ledger/recompute-event.ts` to confirm).
3. Output structured JSON with:
   - Per-line: `line_id`, `order_id`, `sale_date`, `stored_cost_at_sale`, `expected_cost_at_sale`, `delta_vnd`, `associated_event_ids[]`.
   - Per-event: `event_id`, `status`, `source_table`, `source_id`, `item_reference`, `quantity_change`, `unit_cost`, `effective_timestamp`, `visibility_timestamp`, `lag_ms`, `detected_at`, `affected_line_count`, `reviewer` (if not PENDING), `reviewed_at` (if not PENDING).
4. Status breakdown summary: count + delta VND by (PENDING / APPROVED / REJECTED).
5. Coverage check: every one of the 41 lines must have at least one associated event. Flag any line with zero events (would indicate detection mismatch).
