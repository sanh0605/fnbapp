## Hypotheses to test

### H1: Late PO receipts with backdated effective_at

Migration 0014 captures backdating only for inventory-increasing rows >5 minutes backdated with a visibility trigger. PO receipts entered <5 minutes backdated, OR PO receipts entered with `effective_at` set to a past time but created quickly, would not be captured.

**Test**: For each of the 71 lines, query `Stock_Ledger` rows where:
- `source = 'PO_RECEIPT'`
- `item_reference` IN (BTP-002 base ingredients from current recipe)
- `created_at > sale_time` AND `effective_at <= sale_time`

Flag any matches. Compare receipt unit_cost at sale_time vs current replay.

### H2: Late production yields for shortfall BTPs

If BTP-002 yield rows are inserted after a sale but with effective_at <= sale_time, MAC replay would use a different BTP cost than write-time.

**Test**: Query `Stock_Ledger` rows where:
- `source = 'PRODUCTION_YIELD'`
- `item_reference` IN ('BTP-002', 'BTP-009', 'BTP-011', 'BTP-012')
- `created_at > sale_time` AND `effective_at <= sale_time`

### H3: BTP shortfall fallback formula divergence

The MAC engine has a fallback for BTP shortfall (partial base-ingredient decomposition). If the fallback uses different MAC inputs at write vs replay (e.g., latest MAC vs windowed MAC), drift is structural.

**Test**: Read `lib/mac-cogs.ts` and `lib/inventory-consumption.ts`. Identify the shortfall fallback formula. Reproduce one post-cutoff line's write-time inputs exactly. Compare to replay. Document divergence, if any.

### H4: Recipe or batch_yield mutation after sale

Recipes for PROD-006 or BTP-002 may have been edited after sale. The line's `recipe_snapshot_json` should preserve the sale-time recipe, but if the engine ignores snapshot for shortfall decomposition, mutation would cause drift.

**Test**: For each line, parse `recipe_snapshot_json` and compare to current `Recipes` row. Flag any divergence. Cross-check whether the MAC replay uses snapshot or current recipe for BTP shortfall decomposition.

### H5: Operator workflow concentration

If the 71 lines concentrate on one operator, one shift, one supplier, or one POS terminal, the cause may be workflow-specific (e.g., end-of-day PO entry habit).

**Test**: Group the 71 lines by:
- `Orders_V2.created_by` (operator)
- UTC hour of sale_time
- Day of week
- Supplier of related PO receipts

Flag any concentration >30% in one bucket.

### H6: Visibility trigger threshold gap

Migration 0014's 5-minute threshold may miss common-case backdating. If operators routinely enter receipts 3-4 minutes backdated (e.g., after a delivery), the trigger would not fire.

**Test**: Query all `PO_RECEIPT` rows for BTP-002 base ingredients in July 2026. Compute the distribution of `(created_at - effective_at)` intervals. Flag if the distribution clusters at 1-4 minutes (under-trigger zone).

### H7: BTP-002 stock state at sale time vs replay

The BTP_SHORTFALL pattern means BTP stock was insufficient at sale time. If BTP-002 stock state at sale time was X (causing shortfall) but replay computes state Y (no shortfall, or different shortfall), the decomposition differs.

**Test**: For each line, replay BTP-002 stock state at `sale_time` using ledger rows with `effective_at <= sale_time` (write-time view). Compare to current replay state. Document divergence.
