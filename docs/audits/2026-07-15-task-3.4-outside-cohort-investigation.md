# Task 3.4 Outside-Cohort MAC Drift Investigation

Date: 2026-07-15
Owner: Codex
Scope: 224 live MAC mismatches outside the fixed 170-line audit baseline
Contract: read-only production investigation; no database rows were written.

## Executive conclusion

At cohort capture, the live population reproduced exactly: 354 mismatch lines
split into 130 inside the locked baseline and 224 outside it. The 170 database
locks match the 170 frozen source IDs exactly, and none of the 224 outside IDs
overlaps the baseline. There is no locked-versus-outside cohort classification
bug.

During final verification, the live audit advanced to 396 mismatches: the same
130 locked lines plus 266 outside lines. All original 224 outside IDs remained
mismatched and 42 new outside IDs appeared. The script therefore reused the
frozen 224 IDs already captured in the Task 3.4 artifact instead of silently
expanding this task's scope. The 42 new lines are additional forward-drift
evidence and await Claude review with the other forward-looking findings.

Every outside line currently replays through a `BTP_SHORTFALL` path. All 224
signed deltas are negative, totaling -243,917 VND. In this audit,
`delta = expected_cost - stored_cost`; negative therefore means stored COGS is
higher than the current replay, not under-stored.

The final disjoint classification is:

No outside line is an automatic recovery candidate. The 41 causal backdated

investigation.

## Method

`scripts/investigate-task-3.4-outside-cohort.ts` runs the same
`auditMacCogsDrift` replay used by the baseline audit, then subtracts the 170
IDs in both the frozen artifact and `audit_baseline_locks`. It asserts:

1. The source artifact and database each contain the same 170 lock IDs.
2. Current live mismatches split into exactly 130 locked and 224 outside.
3. Outside/source overlap is zero.
4. Every outside line receives exactly one final classification.
5. The final bucket counts sum to 224.

If a later live run discovers more outside mismatches, the script requires the
existing 224-line artifact, verifies that all captured IDs are still present,
and classifies only that frozen cohort. It reports the new IDs separately and
does not fold them into Task 3.4.

For each outside line, the script records sale time, product and variant IDs,
stored and replayed COGS, signed delta, direct and shortfall BTP IDs, consumed
items, purchase-recovery counterfactual, precise/proxy backdated matches,
write-time visibility evidence, migration markers, and order-edit events.

Production access uses service-role credentials because migration 0014 revokes
anon/authenticated access to `backdated_ledger_events`. The script exposes only
a local paginated `SELECT` helper and contains no insert, update, delete, or RPC
call. Its only write is the new local JSON artifact.

## Clarification: 95 raw backdated matches

The first sale-time-only pass found 95 lines whose consumed items intersect a
precise Task 3.2 PO window:

```text
effective_timestamp <= sale_time <= purchase_order.created_at
```

That condition is necessary but not sufficient for causality on migrated
history. A migrated order may preserve an old sale timestamp while being
written after the PO was already visible. The refined test compares PO
visibility with the actual line-write proxy from `Order_Events`:

```text
causal only when purchase_order.created_at > order write visibility time
```

The 95 lines split as follows:

| Evidence subtype | Lines | Delta | Interpretation |
| --- | ---: | ---: | --- |
| Causal backdated exposure | 41 | -43,809 VND | PO was still invisible when the order line was written |
| Legacy migration correlation | 54 | -62,602 VND | Migrated line was written after the matched PO was already visible |
| **Raw sale-window fingerprint** | **95** | **-106,411 VND** | Must not be treated as one recovery cohort |

The 41 causal lines contain two real subgroups:

- 22 V1-migrated lines were written around 2026-06-20, before the matched
  later-created PO became visible.
- 19 non-migrated live lines were written before the matching PO visibility
  timestamp.

The 54 correlation-only lines are all migration-marked. They retain historical
sale timestamps inside old PO effective/visibility windows, but their migration
write occurred after those POs existed. They are not evidence that hidden
backdating changed their stored COGS and are classified by temporal scope as
`PRE_BASELINE_WINDOW`, not `BACKDATED_LEDGER_LIKE`.

This distinction reduces the defensible Task 3.2 review queue from the raw 95
sale-window matches to 41 write-time-hidden matches. It still does not make
those 41 safe for automatic recompute; the evidence identifies exposure, not
an approved accounting restatement.

## Locked-cohort replay shift

The E3 verification artifact recorded +120,716 VND for the 130 retained locks.
That number is the reviewed lock-plan delta. The current live replay of the
same 130 locked mismatch IDs is +102,621 VND, a -18,095 VND expected-replay
shift.

| Measure | Delta |
| --- | ---: |
| Frozen 130-lock reviewed delta | +120,716 VND |
| Current replay of the same 130 IDs | +102,621 VND |
| Replay movement | -18,095 VND |

No locked stored `cost_at_sale` value was changed by this investigation. The
movement is in recomputed expected cost as the live effective ledger evolves.
This is another reason the fixed lock artifact and a live replay must not be
treated as interchangeable accounting snapshots.

The earlier implied outside delta of approximately -262,013 VND mixed the
current full-audit total with the frozen +120,716 VND lock delta. The coherent
current line-level reconciliation is:

```text
+102,621 VND locked + (-243,917 VND outside) = -141,296 VND
```

The full audit headline is -141,297 VND; the 1 VND difference is the same
sub-threshold/line-rounding distinction already documented by Task 3.3.

## Hypothesis verdicts

### H1: Post-cutoff lines are new live backdating

**Verdict: not supported by durable backdating events.**

All 71 post-cutoff lines were checked against `backdated_ledger_events` using
consumed item, effective timestamp, visibility timestamp, and sale time. Zero
lines match a durable event. They also have zero precise historical Task 3.2
matches. Backdating is therefore not the evidenced cause of this post-cutoff
group.

### H2: Post-cutoff lines are new engine drift

**Verdict: rejected.**

Git history since 2026-07-02 across `lib/mac-cogs.ts`,
`lib/inventory-consumption.ts`, `app/pos/actions.ts`, and
`app/admin/orders/actions.ts` contains one matching commit, `5b315eb`. It only
changes snapshot-first product/variant display-name lookup in an admin read
path. It does not alter MAC, inventory consumption, or `cost_at_sale` writes.

### H3: Pre-cutoff lines are a baseline scope gap

**Verdict: confirmed as a frozen-snapshot gap, not a divergent selection
predicate.**

The 153 lines dated on or before the cutoff have zero ID overlap with the
frozen 170. The original and current audits use the same completed,
non-superseded-order and mismatch-threshold criteria; neither has a sale-date
filter. The JSON is a point-in-time mismatch list, so lines can enter the live
population later when effective ledger inputs change.

Twenty-three outside lines fall in the observed frozen-baseline date span.
One has causal backdated evidence and is classified accordingly; the other 22
are `BASELINE_SELECTION_GAP`. Task 3.5 should make the audit explicitly
cohort-aware and prevent replacement of the frozen artifact, but there is no
locked/outside ID misclassification in this run.

### H4: Pre-baseline-window history is material

**Verdict: confirmed.**

The earliest mismatch represented in the frozen 170-line baseline is
2026-06-25T08:14:33.946Z. The current outside population contains 130 lines
before that timestamp totaling -150,082 VND. Forty-seven are before
2026-06-01 and total -52,166 VND, exceeding the proposed 20-line materiality
threshold.

After causal precedence, 40 of the 130 pre-window lines enter the backdated
review bucket and the other 90 remain `PRE_BASELINE_WINDOW`. Extending a
recovery baseline is not recommended without a separate accounting decision;
the current task only establishes scope.

### H5: Sign asymmetry is concentrated

**Verdict: confirmed.**

All 224 outside deltas are negative. The concentration is not even:

| Product | Lines | Delta | Share of absolute outside delta |
| --- | ---: | ---: | ---: |
| `PROD-006` | 126 | -140,639 VND | 57.7% |
| `PROD-023` | 56 | -75,162 VND | 30.8% |
| Other products | 42 | -28,116 VND | 11.5% |

Shortfall BTP concentration is stronger:

| BTP | Lines | Delta |
| --- | ---: | ---: |
| `BTP-002` | 183 | -216,754 VND |
| `BTP-009` | 41 | -27,163 VND |
| `BTP-011` | 4 | -6,655 VND |
| `BTP-012` | 4 | -3,800 VND |

Some lines contain more than one shortfall BTP, so BTP row counts and deltas
are not additive. The dominant mechanism is nevertheless clearly centered on
`BTP-002` and the two products above.

### H6: BTP shortfall recurs outside the baseline

**Verdict: confirmed and dominant.**

- Outside cohort: 224/224 lines, 100%, -243,917 VND.
- Frozen baseline: 132/170 lines, 77.6%.
- Post-cutoff: 71/71 lines through 2026-07-14.

This is evidence of an active forward-looking drift pattern. It does not by
itself prove that the current MAC or consumption engine is wrong; it proves
that stored historical values continue to diverge from later live replay in
the BTP-shortfall path. Per the review decision, this report records the
finding and waits for Claude before opening a separate forward-drift task.

### H7: Edit-order side effects

**Verdict: partial correlation, not causal proof.**

Only 2/224 lines have an `EDITED` event after the preserved sale timestamp;
their combined delta is -2,522 VND. Current `editOrderV2` explicitly recomputes
and pins `cost_at_sale` at the original sale time before inserting the edited
version. Two correlated rows cannot explain the 224-line population, and the
retained data does not show that their COGS was left unpinned.

## Recovery and follow-up recommendations

### Automatic recompute

None. Zero lines reproduce the exact `STK-014`/`STK-018`/`STK-019` purchase
cost recovery counterfactual within the reviewed 2026-06-30 to 2026-07-02
window. No E4-style purchase-cost candidate exists, and no single outside line
exceeds the 10,000 VND escalation threshold.

### Task 3.2 review path

Review the 41 write-time-hidden `BACKDATED_LEDGER_LIKE` lines through the
existing backdated-ledger accounting workflow. Do not include the 54
migration-correlation false positives and do not auto-recompute the 41 without
an approved restatement plan.

### Task 3.5 dependency

Use the 22 `BASELINE_SELECTION_GAP` lines as evidence that the baseline audit
must separate frozen-cohort verification from live-population discovery. The
fix should also prevent `audit-mac-drift-baseline.ts` from overwriting the
reviewed 170-line artifact.

### Forward drift

The 71 post-cutoff BTP-shortfall lines require forward detection and root-cause
work, not historical recovery under Task 3.4. Wait for Claude review before
opening or implementing that task.

### Historical scope

Keep the 90 final `PRE_BASELINE_WINDOW` lines as documented historical audit
drift unless the owner explicitly approves extending the accounting baseline.
The 47 pre-June lines make the scope material, but materiality alone is not a
causal recovery theory.
