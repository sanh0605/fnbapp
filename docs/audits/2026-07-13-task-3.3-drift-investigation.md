# Task 3.3 MAC Drift Investigation

Date: 2026-07-13
Owner: Codex
Scope: fixed 170-line baseline from `2026-07-09-mac-drift-baseline-lines.json`
Contract: read-only production investigation; no database rows were written.

## Executive conclusion

The 170 baseline lines still reproduce exactly: all 170 current replay costs
match the expected costs stored in the baseline artifact. The investigation
therefore analyzed a stable historical discrepancy rather than live-data
noise.

The lines split into three disjoint recovery buckets:

| Bucket | Lines | Signed baseline delta | Absolute delta | Conclusion |
| --- | ---: | ---: | ---: | --- |
| Purchase-cost recovery on 2026-07-02 | 40 | -933 VND | 933 VND | Confirmed by exact counterfactual replay |
| Previously detected backdated ledger exposure | 34 | +1,762 VND | 2,906 VND | Confirmed exposure; no additional <=5-minute cases |
| Missing write-time provenance | 96 | +118,954 VND | 119,038 VND | Exact cause is no longer falsifiable from retained data |

The per-line signed sum is +119,783 VND. The baseline headline is +119,782
VND because the audit total and individually rounded line totals differ by 1
VND.

The original 97.6% remainder cannot be assigned honestly to one migration,
recipe, or engine commit. The dominant 96-line group was written from
2026-06-26 11:21:35Z through 2026-06-29 02:23:02Z, but `Stock_Ledger` has no
durable `recorded_at`/visibility timestamp and the order lines do not store the
ledger snapshot or valuation-version hash used when `cost_at_sale` was written.
Current data can reproduce today's replay, but cannot reconstruct the exact
write-time input set for those 96 lines.

## Method

`scripts/investigate-task-3-3-drift.ts` loads the fixed 170 line IDs, reads the
operational tables without mutation helpers, and runs these counterfactuals:

1. Current deterministic MAC replay.
2. Sale-time semi-product recipe replay.
3. Pre-`d23211f` last-recipe-wins replay.
4. Historical FIFO and direct-FIFO variants.
5. Replay with recent consumption rows excluded across 1, 5, 15, 30, 60,
   180, 360, 720, and 1,440 minute windows.
6. Replay with the three 2026-07-02 purchase-cost corrections restored to
   their prior unit costs.
7. Precise PO backdating cross-reference using effective time, PO creation
   time, sale time, and consumed item IDs.

The script also compares line recipe snapshots to current product/modifier
recipes, compares sale-time and current semi-product recipes, records relevant
Git history, and emits per-line cost contributions to the JSON artifact.

## Hypothesis verdicts

### H1: Historical backfill or migration artifact

**Verdict: rejected as the primary cause.**

Only 2/170 lines have `MIGRATED`/`IMPORTED` events or non-empty
`migration_notes`. Their combined baseline delta is -54 VND. The other 168
lines lack migration markers, so marked backfill orders cannot explain the
dominant positive drift.

The initial hypothesis remains limited by metadata quality: unmarked manual
imports cannot be proven absent. However, no retained artifact connects such
an import to the 96-line dominant group.

### H2: Backdating beyond Task 3.2 detection scope

**Verdict: rejected beyond the existing precise PO scope; historical
`STOCK_ADJUST`/`PRODUCTION_YIELD` remains structurally inconclusive.**

- Existing Task 3.2 evidence still matches 34 baseline lines.
- Sum of absolute matched drift remains 2,906 VND.
- A zero-threshold scan found the same 34 lines.
- Matches with lag at or below the 5-minute trigger threshold: 0.

Therefore, relaxing the five-minute threshold does not discover more precise
PO cases. Historical `STOCK_ADJUST` and `PRODUCTION_YIELD` rows still lack an
independent visibility timestamp, so additional backdating through those types
cannot be proven or rejected retrospectively.

### H3: Order-line recipe snapshot drift

**Verdict: rejected as a replay cause.**

Thirty-six lines have a snapshot that differs from the current product or
modifier recipe. This is not causal for this audit: `auditMacCogsDrift` parses
each line's stored `recipe_snapshot_json`; it does not substitute the current
product/modifier recipe. Recipe snapshots are behaving according to the
snapshot contract.

### H4: MAC engine code drift

**Verdict: partial evidence on two lines, rejected as the primary cause.**

Relevant history includes the MAC cutover (`8236818`), shared consumption
refactor (`1cae265`), deterministic recipe selection (`d23211f`), and atomic
compact POS state (`12dd2db`). The investigation replayed the material engine
variants rather than relying only on commit dates:

- Current MAC exact matches to stored cost: 0 lines.
- Legacy last-recipe-wins MAC exact matches: 0 lines.
- Full FIFO exact matches: 0 lines.
- Direct FIFO exact matches: 2 lines.
- Recent-consumption visibility-window exact matches: 0 lines.

Two direct-FIFO matches are insufficient to attribute the 170-line baseline to
FIFO. No tested historical engine explains the primary drift.

### H5: Semi-product recipe or batch-yield change

**Verdict: recipe change rejected; batch-yield history inconclusive.**

- Lines whose sale-time semi-product recipe differs from current: 0.
- Lines whose stored cost is restored by sale-time recipe replay: 0.
- The 2026-07-01 immutable snapshot shows the same 13 `batch_yield` values in
  Google Sheets and Supabase.

`Semi_Products` stores only one mutable row per ID, so it does not preserve
earlier yield values. There is no positive evidence of a yield change, but the
schema cannot prove that no edit occurred before the first immutable snapshot.

### H6: Peak-date-specific data event

**Verdict: concentration confirmed; standalone peak-date event rejected.**

The 2026-06-26 peak remains 25 lines / +86,232 VND. Ledger activity on that
UTC date consists of 253 `SALES_CONSUME` and 54 `EDIT_REVERSAL` rows; no
positive inventory event on that date explains the peak.

The only semi-product recipe created on the date is `RC-029` for `BTP-013`
(Trứng luộc). The high-delta peak lines instead predominantly use BTP-002,
BTP-003, BTP-008, BTP-010, and BTP-011. Standalone-topping variant recipes
`REC-071` through `REC-077` were created later and do not explain the earlier
peak orders.

### Additional finding: 2026-07-02 purchase-cost recovery

**Verdict: confirmed for 40 lines, but not the dominant cause.**

The reviewed recovery changed three historical receipt unit costs:

| Ledger | Item | Before | After |
| --- | --- | ---: | ---: |
| `STK-014` | `ING-032` | 69 | 68.541667 |
| `STK-018` | `ING-012` | 98 | 98.412698 |
| `STK-019` | `ING-022` | 20 | 19.6 |

Restoring those three old values in memory makes 40 baseline lines match their
stored `cost_at_sale` exactly. Their net baseline delta is -933 VND. This is a
clear and safe causal bucket, but it is small and opposite in sign to the
dominant +118,954 VND unresolved group.

## Why the remaining 96 lines are inconclusive

The investigation found useful fingerprints—for example, many +607 VND lines
share 40 units of `ING-003`, consistent with a historical valuation-rate
difference—but no retained record identifies when the relevant ledger state
became visible to the write path. Common Google Sheets/Supabase ledger rows in
the 2026-07-01 snapshot preserve item, type, effective time, and unit cost;
their material fields do not show a migration mutation that explains the
group.

The missing evidence is the write-time state itself:

- `Stock_Ledger.created_at` is an effective timestamp, not a durable insertion
  or visibility timestamp.
- Historical adjustment/yield rows have no independent source-record time.
- Order lines store the result (`cost_at_sale`) and recipe snapshot, but not the
  valuation method/version, input cutoff, or ledger/source hash.
- No immutable full ledger snapshot exists at each affected checkout.

Assigning these lines to FIFO, recipe drift, or a specific migration would
therefore overstate the evidence.

## Task 3 recovery recommendation

### Safe to recompute

The 40 purchase-cost-recovery lines are the strongest automatic recovery
candidate. The cause is explicit, reversible, and reproduced exactly by the
counterfactual. Recompute only from the reviewed baseline list and retain the
existing lock/recovery audit controls.

### Manual review before recompute

The 34 backdated-ledger lines should remain in the Task 3.2 review workflow.
They have precise PO visibility evidence, but historical recompute is an
accounting decision and should not be mixed silently with the 40-line purchase
cost bucket.

### Accept and lock as audit drift

The 96 provenance-gap lines should not be auto-recomputed under a claimed root
cause. Recommended default: keep stored historical COGS, lock the reviewed
baseline rows, and document +118,954 VND signed drift against the current replay
contract. A bulk recompute remains possible only as an explicit accounting
policy choice, not as a repair backed by reconstructable write-time evidence.

## Schema recommendations before future historical recovery

1. Separate `effective_at` from immutable `recorded_at` on every stock-ledger
   row.
2. Persist valuation metadata on order lines: method, engine version, ledger
   cutoff, and a source/input hash.
3. Version `Semi_Products.batch_yield` instead of mutating a single row.
4. Preserve an independent source-created timestamp for stock adjustments and
   production yields.
5. Continue atomic checkout so COGS calculation and ledger insertion share one
   consistent database snapshot.
